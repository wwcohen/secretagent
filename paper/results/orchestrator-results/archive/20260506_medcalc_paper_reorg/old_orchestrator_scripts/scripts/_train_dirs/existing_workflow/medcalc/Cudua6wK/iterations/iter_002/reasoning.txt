An analysis of the failure cases reveals three specific failure modes causing the train-eval gap:

1. **Date Math & Complex Unit Conversions**: The LLM struggles fundamentally with date arithmetic (`Estimated Due Date`, `Date of Conception`) and complex formula matching (e.g., matching multiple drug dosages for `Morphine Milligram Equivalents`, or precise lab conversions like `Free Water Deficit`). The pure Python regex extraction (`calculators.calculate`) handles these deterministically, but the pipeline forces them through the LLM.
2. **Artificial Validation Failures**: For scoring calculators with many optional boolean flags (like `CHA2DS2-VASc` or `Wells`), the LLM sometimes correctly extracts the `True` conditions but omits the `False` ones from the JSON. The validation function `_validate_extracted_values` had a hardcoded rule that if `len(optional) >= 4`, missing any optional field forced a fallback to the baseline `simulate_medical_value`. This baseline hallucinates wildly. We need to trust the partial extraction and default the missing flags to `False`.
3. **Over-Inferring Acute Symptoms as Baseline History**: In `analyze_scoring_conditions`, the prompt encourages inferring conditions. For baseline risk scores like `CHA2DS2-VASc`, the LLM incorrectly scores the patient's *current acute presentation* (e.g., an acute stroke) as a *past medical history* condition, inflating the score.

### The Fix:
1. **Targeted Python Fast-Path**: Intercept a strictly whitelisted set of "exact/math" calculators right after `identify_calculator` and route them to the deterministic `calculators.calculate`. If it succeeds, return the result. This isolates the LLM from tasks it is physically incapable of (date math) without overriding its superior clinical reading for the rest.
2. **Robust Validation**: Removed the `len(optional) >= 4` check in `_validate_extracted_values`. We now only force a fallback if the LLM completely missed the entire optional section (`extracted_optional == 0`).
3. **Targeted Prompt Tweak**: Added a specific instruction to `analyze_scoring_conditions` reminding the LLM to separate baseline history from acute presentation.
