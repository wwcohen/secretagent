An analysis of the failure cases uncovers a fundamental flaw in how the orchestration logic queries the LLM: **evaluating the suspects independently causes catastrophic collisions.**

1. **The MuSR Pattern**: In almost every case, exactly ONE suspect is the "main" focal point of the investigation (many mentions), while the other is a "ghost" suspect (barely mentioned). 
2. **The "Ghost" Hallucination**: When asked to evaluate the ghost suspect, the LLM hallucinates evidence or gets confused because the narrative barely discusses them. 
3. **The Red Herring**: The narrative always heavily implicates the main suspect. The entire puzzle hinges on whether the main suspect committed the crime, or if there is a subtle clue proving them innocent (e.g., an Alibi, a Disguise/Framing, Fake Evidence like paint instead of blood, or an Obsessed Third Party). 
4. **The Tie-Breaker Failure**: If the LLM correctly realizes the main suspect has an alibi and returns `0` for both suspects, the current fallback logic returns the suspect with the *most* mentions — effectively returning the main suspect anyway, and failing all Alibi/Disguise cases!

### The Targeted Fix:
We will **stop evaluating the ghost suspect**. We only need ONE LLM call.
We will use the mention counts to identify the "main" suspect, and then ask the LLM a strictly binary question: *Is this main suspect the actual killer, or are they an innocent red herring?*
If the LLM finds definitive exculpatory evidence (Alibi, Disguise, Fake Evidence, etc.), it outputs `0`, and we immediately return the *other* suspect. Otherwise, we return the main suspect. 

This directly maps to the deductive logic of the dataset, fixes the `[0, 0]` tie-breaker bug, eliminates hallucinations on the ghost suspect, and **cuts the LLM cost in half**.
