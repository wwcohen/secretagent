1. What SPECIFIC failure pattern did you identify?
The pipeline has two primary sources of error:
A) ToM logic failures in the initial LLM nodes (hallucinating distractions).
B) Downstream extraction failures where `extract_index` either crashes or returns `-1` because the LLM generated a valid sub-location (e.g., "the set" or "corner shelf") that doesn't strictly match the choices ("beach set" or "bakery"). 

2. How many of the shown failures share this pattern?
While many failures are ToM hallucinations (which are risky to fix via blanket prompt engineering as seen in Iters 1-3), at least 2 out of the 23 explicitly trace back to `extract_index` returning `-1` (Ex073, Ex068). Additionally, Iter 2 notes that the `extract_index` occasionally crashes.

3. What is the MINIMAL change that fixes this pattern?
- **Avoid touching the `timeline` and `knowledge` prompts**, as attempting to "fix" the LLM's Theory of Mind reasoning has consistently caused regressions on the 51 working cases.
- **Append the choices constraint gently to the final `enhanced_narrative`** so `_extract_belief_location_impl` knows the allowed vocabulary and is less likely to output unmapped locations.
- **Wrap `extract_index` in a `try...except` and provide a robust Python fallback** ONLY when `extract_index` fails or returns `-1`. This fallback uses substring matching and token overlap to correctly map semantic variations. 

4. Why will this change NOT break currently working cases?
Unlike Iter 4 which completely *replaced* `extract_index` (likely breaking cases where it performed custom logic), this change **preserves the exact existing call**. The fallback logic only executes if the pipeline was already guaranteed to fail (i.e. crashing or returning `-1`). Returning a fuzzy match or a default `0` in these scenarios strictly increases expected accuracy with zero risk to correctly mapped cases. The vocabulary hint is also safely isolated to the final JSON structuring step.
