An analysis of the pipeline's failures reveals two core issues preventing the L4 workflow from maximizing its accuracy:

1. **Bypassed Orchestration:** The auto-generated `calculate_medical_value_orchestrated_seed` entry point was hardcoded to return `simulate_medical_value(patient_note, question)`, completely bypassing the advanced L4 orchestration (`workflow`) and the highly deterministic pure Python `distilled_workflow`. This caused the benchmark to run solely on the raw LLM `simulate` tool, causing numerous math arithmetic errors.
2. **Aggressive Validation Sabotaging Complex Scores:** The `_validate_extracted_values` function had an aggressive check that marked extraction as `invalid` if >=4 optional fields were missing or not extracted. However, for complex scoring systems (like CHA2DS2-VASc), conditions not mentioned in the text are meant to default to `False/0`. Rejecting them forced a fallback to the unreliable `simulate_medical_value` instead of using the deterministic Python calculator with appropriate default values.
3. **Unit Parsing Errors:** The LLM often extracts numeric values with units (e.g., `{"creatinine_mg_dl": "1.2 mg/dL"}`). The float parser crashed on these strings, causing the computation to fail.

### Strategy
- Point `calculate_medical_value_orchestrated_seed` to a smart 3-tier cascade:
  1. Pure Python `calculate()` (Perfect for simple equations and exact matching, safely resolving dates).
  2. L4 `workflow()` (LLM two-stage extraction + Python math for complex scores).
  3. LLM `simulate_medical_value()` (Ultimate fallback).
- Relax the `_validate_extracted_values` logic so missing optional fields don't invalidate the payload. The Python `compute_direct` function will safely map these missing fields to their default `False/0` values.
- Enhance the validation dictionary cleaner to strip units using robust regex, and seamlessly parse textual male/female indicators.
- Adjust the return type hints of the workflows to `Any` so that Python can cleanly pass through String returns (dates, tuples) generated by the pure Python calculators without `secretagent` enforcing float conversion.
