## STRATEGY — READ THIS CAREFULLY

Your changes are evaluated on ALL cases, not just the failures shown above.
A change that fixes 5 failures but breaks 15 working cases is a NET REGRESSION.
This has happened in previous iterations — learn from it.

**CONSERVATIVE APPROACH REQUIRED:**
- Changes to docstrings/prompts affect ALL cases that go through that code path.
  If you add instructions that help edge cases but confuse the LLM on common
  cases, accuracy drops. Be surgical.
- Changes to Python logic (extraction, validation, computation) are SAFER because
  they only fire when their conditions are met.
- Prefer adding NEW conditional branches or fallback logic over modifying
  existing working code paths.
- If a previous iteration was rolled back, DO NOT try a more aggressive version
  of the same idea. Try a completely different approach or a much smaller change.

**PRIORITIZE THESE CHANGE TYPES (safest to riskiest):**
1. Fix a bug in Python computation logic (zero risk to other cases)
2. Add a new fallback/recovery path for a specific failure pattern
3. Improve validation to catch and fix a category of extraction errors
4. Add a small, specific instruction to a docstring (low risk if well-scoped)
5. Rewrite extraction/prompt logic (HIGH RISK — only if you are very confident)

**AVOID:**
- Rewriting the extraction prompt wholesale — this breaks working cases
- Adding many instructions at once — impossible to know what helped vs hurt
- Changing unit conversion logic broadly — most values are already correct
