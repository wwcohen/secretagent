An analysis of the failure cases reveals that the LLM has a relatively high success rate overall, but its main point of failure lies in the **routing task (Hamiltonian Path generation)**. Specifically:
1. It frequently hallucinates unidirectional flights as bidirectional (e.g. `from Tallinn to Vilnius`), using routes that don't exist.
2. It struggles to simultaneously respect all meeting/wedding time windows and stay durations, which leads to mathematically invalid timelines or dropped cities.

**Strategy:**
A previous iteration attempted to completely replace the LLM routing component with a pure Python DFS and failed because the hardcoded logic was forced on *every* case, breaking existing working examples. 
We will take a **much safer, targeted approach** by introducing a **Fallback Validation Pattern**:
1. We preserve the LLM's `find_valid_route` workflow so working cases are **untouched**.
2. We add a `verify_route` function that uses a simple regex to perfectly parse the directed graph from the `problem_text` and validates the LLM's route against all constraints.
3. If and **ONLY IF** the LLM generates an invalid route, we fire a `python_dfs_fallback` to calculate the exact, correct Hamiltonian path. If the DFS succeeds, we substitute the corrected route before formatting.
4. We also apply a surgical formatting instruction to `build_trip_plan` to perfectly align its first line with the golden eval standard (`Arriving in {City}...`).
