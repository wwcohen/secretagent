An analysis of the execution traces reveals three core issues holding back the L4 pipeline's accuracy:

1. **Brittle Early Returns:** 18 out of the 51 failures showed *no LLM execution trace whatsoever*. This means the pure Python regex `calculate()` function fired, extracted the wrong numbers, and confidently returned a wrong answer, completely bypassing the highly accurate L4 LLM extraction workflow. 
2. **Type Coercion on Dates:** Date-based calculations (like Estimated Due Date) return strings or tuples. If they fall back to the generic `simulate_medical_value`, `secretagent` attempts to parse them as a `float`, causing crashes or returning NaN.
3. **LLM Output Formatting:** Sometimes the LLM wraps JSON in markdown block ticks (e.g., ````json { ... } ````), causing the `isinstance(result, dict)` check to fail and silently drop the LLM's excellent reasoning. Furthermore, it occasionally hallucinates "vascular disease" for conditions like small-vessel disease in the CHA2DS2-VASc score.

### Strategy
- **Demote Pure Python:** We swap the order in `calculate_medical_value_orchestrated_seed`. We now run the robust LLM-orchestrated `workflow()` *first*. The pure Python `calculate()` is kept strictly as a fallback. 
- **Isolate Date Calculators:** We add a fast check for date-related questions and route them to a specialized `simulate_date_value` endpoint with a custom return parser that safely handles strings and tuples without float coercion.
- **Harden LLM Parsing:** We inject a `re.search(r'\{.*\}')` fallback to extract JSON dictionaries even when the LLM wraps them in markdown block strings.
- **Clarify Complex Scores:** We add a small, targeted clarification to the reasoning prompt about the strict definition of vascular disease and advise against confusing head circumference with weight.
