An analysis of the failure cases and previous iterations reveals three primary issues:

1. **Unwanted Lab Value Conversions**: The LLM is "too smart" when extracting lab values. For example, in the CKD-EPI calculator, it converted `120 mmol/L` of creatinine into `1.35 mg/dL`, and in Calcium Correction it multiplied the ionized calcium. The benchmark's reference computation often uses the raw numbers exactly as written in the text. We need to explicitly instruct the extraction prompt to pull the raw numeric values for labs without converting them.
2. **Schema Mismatch for Dynamic Calculators**: Calculators like Dates (Estimated Due Date, Date of Conception, Gestational Age) and drug equivalents (Steroid Conversion, Morphine Milligram Equivalents) require dynamic inputs (e.g., date math from LMP, or arbitrarily named target steroids/opioids) that do not fit neatly into the fixed JSON signature expected by `compute_calculation`. Passing these to the L4 Python compute results in severe math errors. They should be intercepted and routed to the `simulate_medical_value` fallback, which uses the `FORMULA_REFERENCE` and Chain-of-Thought to solve them flexibly.
3. **Loss of Medical Reasoning on Fallback**: The L4 pipeline has a safety mechanism that routes scoring calculators with $\ge 4$ optional boolean criteria to the `simulate_medical_value` fallback because the JSON extraction drops points. However, when it falls back, the rich medical reasoning generated by `analyze_scoring_conditions` is discarded, forcing the LLM to start from scratch. Passing this reasoning context into the fallback note will drastically improve scoring accuracy for CHA2DS2-VASc, Wells, and others.
