1. **Specific failure pattern**: The orchestrated LLM operates in a zero-shot manner on the main entrypoint (`calculate_medical_value`) and bypasses the induced sub-tools. This results in two major failure modes: 
   a) **Arithmetic errors**: The LLM hallucinates additions for scoring systems (APACHE II, HAS-BLED, etc.) and complex formulas (MDRD, Delta Gap). 
   b) **Date extraction failures**: The pipeline currently converts all outputs to `float`, making it impossible to return expected date strings (`11/24/2005`) or tuples (`6 weeks, 0 days`).

2. **Impact**: Almost all 42 shown failures fall into these two categories. Date calculations failed 6 times, and math/summation errors on scoring systems and formulas accounted for the rest.

3. **Minimal fix**: 
   - **Fixing Date formatting & Math evaluation securely**: Modify the `_extract_number` parser to natively recognize `<math>...</math>` tags (evaluating them with a safe, rigid Python AST parser) and `<date>...</date>` tags (returning the exact string or tuple expected). 
   - **Fixing the LLM behavior**: Update the `calculate_medical_value` docstring (which serves as the main prompt) to explicitly instruct the LLM to output its math expressions in `<math>` tags and dates in `<date>` tags. Additionally, embed the specific scoring criteria for the failing tests directly into the main docstring so the LLM has zero-shot access to the rules it needs.

4. **Why it's safe**: 
   - It avoids brittle Python routing (which caused regressions in Iter 1).
   - If the LLM doesn't output `<math>` or `<date>` tags, or if the math expression contains a syntax error, the `_extract_number` function cleanly falls back to the exact regex logic that currently achieves 61.8% accuracy. 
   - The original tools are kept perfectly intact for context ingestion by the orchestration framework.
