1. **Specific failure pattern**: 
   - **Knowledge/Hallucination errors**: The LLM frequently zero-shots complex clinical score questions (Caprini, Centor, FeverPAIN, SIRS, PERC, RCRI, Charlson) and formulas (Delta Gap, Free Water Deficit, MDRD, CKD-EPI) directly, but lacks the exact point allocations or coefficients, leading to arithmetic and formula hallucinations.
   - **Date extraction destruction**: For the 6 date/gestational age questions (e.g. `11/24/2005` or `('6 weeks', '0 days')`), the LLM answers correctly, but the legacy `_extract_number` function forcibly strips non-digit characters and casts the string to a `float` (turning `11/24/2005` into `11242005.0`), completely ruining the result.
   
2. **Impact**: Almost all 42 shown failures are either date extraction errors or zero-shot hallucination/arithmetic errors on specific calculators. 

3. **Minimal fix**:
   - **Expand Tool Knowledge Base**: Inject concise reference guides containing exact criteria distributions and standard formulas into the docstrings of `apply_clinical_score` and `calculate_clinical_score`. Since the LLM reads the tool docstrings (even when deciding to zero-shot), this completely mitigates the knowledge hallucination without forcing brittle Python routing.
   - **Strict & Safe Date Parsing**: Modify `_extract_number` to safely return date strings (`MM/DD/YYYY`) and gestational age tuples `('X weeks', 'Y days')`. By using `re.finditer` to locate the end-positions of the last date, last tuple, and last float in the string, we can securely return a date/tuple *only* if it appears at the very end of the LLM's response (or inside an `<answer>` tag), perfectly preserving the float fallback for all other cases.
   
4. **Why it's safe**:
   - We avoid rewriting the orchestration loop or forcing keyword routing, which broke working cases in Iter 1 and 4.
   - The position-based date extraction logic ensures that casual mentions of dates/weeks in the patient note won't override a float answer that appears later in the text. 
   - Modifying tool docstrings is purely additive context for the LLM; it doesn't change the execution graph or break working extraction patterns.
