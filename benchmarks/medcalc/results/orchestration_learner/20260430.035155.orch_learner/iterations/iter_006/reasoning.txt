1. **What SPECIFIC failure pattern did you identify?**
   The pipeline suffers from two severe issues:
   - **Schema Validation Crashes (3 cases):** The main entry point `calculate_medical_value` is enforced by the Orchestrator to strictly return a `float`. When the LLM attempts to output step-by-step text to explain its reasoning, it fails the JSON schema validation, enters a retry loop, and crashes the pipeline with a `request_limit of 50` exception.
   - **Zero-Shot Calculation Errors (42 cases):** Because the LLM is forced to output *only* a float, it cannot use Chain-of-Thought reasoning. This causes massive inaccuracies when calculating complex, multi-criteria medical scores (like APACHE II, Caprini, HEART, and HAS-BLED). 

   In previous iterations (like Iteration 1), replacing `calculate_medical_value` with a `direct` Python implementation that called a CoT tool failed because (a) the prompt forced ALL questions into a "clinical score" format, failing on simple values (like age), and (b) the `_extract_number` function blindly grabbed the *last* number in the text (`nums[-1]`), picking up random numbers from the explanation instead of the final score.

2. **How many of the shown failures share this pattern?**
   All 45 failures stem from this limitation. Complex scores require step-by-step reasoning, but the framework forces a zero-shot float return. 

3. **What is the MINIMAL change that fixes this pattern?**
   - I will change `calculate_medical_value` to a `direct` Python implementation to completely bypass the Orchestrator's restrictive JSON schema validation, eliminating the 3 crash exceptions.
   - Inside it, I will call a *new* unified tool (`_solve_medical_value_cot`) that takes both `patient_note` and `question`. Its prompt is carefully designed to gracefully handle BOTH complex scores (by explicitly thinking step-by-step) and simple extractions, ensuring we don't regress on simple queries.
   - I will significantly upgrade the `_extract_number` function to prioritize extracting numbers explicitly wrapped in `<answer>` tags, ensuring that intermediate numbers from the CoT reasoning don't pollute the final output.

4. **Why will this change NOT break currently working cases?**
   By ensuring `calculate_medical_value` returns a clean `float`, we satisfy the evaluator's expectations. The new unified prompt explicitly accommodates simple extraction queries without forcing them into a "score calculation" hallucination, preserving the accuracy of the 31 cases that already work. The upgraded `_extract_number` gracefully falls back to the original extraction logic if tags are missing, ensuring robustness.
