1. **Specific failure pattern identified**: The model fails to accurately calculate scores because it hallucinates or misremembers the specific point values and thresholds for complex scores (like Caprini, HAS-BLED, GBS). Furthermore, it occasionally encounters network timeouts or transient framework crashes (TypeValidationError from SecretAgent when it loops) which manifest as "Connection timed out" exceptions.
2. **How many shown failures share this**: All 31 non-timeout failures are caused by incorrect score aggregation/misremembered rules. The 2 timeout failures are caused by network/crash loops.
3. **Minimal change**: Instead of wrapping the entry point in a brittle CoT tool (which Iter 1 proved is unstable due to regex extraction) or dumping all 13 rubrics into the global prompt (which Iter 3 proved causes overfitting and cross-contamination), we change `calculate_medical_value` to a Python `@implement_via('direct')` function. This function uses a dictionary of specific rules and **dynamically injects ONLY the relevant rubric** into the question for the requested score, keeping the prompt lean and focused. It then delegates to a new `_simulate_calculate_medical_value` tool that cleanly utilizes the framework's native `float` return extraction. Additionally, a simple 3-attempt retry loop is wrapped around the simulation call to catch and resolve the transient timeouts.
4. **Why this won't break working cases**: For any score not explicitly defined in the dictionary (or if no match is found), the exact original zero-shot logic executes with no modifications, completely eliminating regression risks for unaffected scores. We preserve the framework's strict native `float` parsing while solving the memory/attention bottleneck uniquely per question.
