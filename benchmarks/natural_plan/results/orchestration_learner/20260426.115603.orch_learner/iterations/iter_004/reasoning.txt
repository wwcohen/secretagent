An analysis of the failure cases highlights three distinct, fixable patterns that are degrading accuracy:

1. **Date Math / Formatting Errors (High Frequency):** The LLM hallucinates overlapping flight days and fails to use the exact `Arriving in {City}` prefix for the first day, causing perfectly valid routes to fail string-matching evaluation.
2. **Hallucinated Bidirectional Flights (Medium Frequency):** `parse_trip_constraints` extracts unidirectional flights (e.g. `from Tallinn to Vilnius`) as bidirectional, feeding a corrupted graph to `find_valid_route`.
3. **Dropped Cities / Ignored Time Windows (Medium Frequency):** The LLM occasionally drops cities or slightly offsets the strict time windows when generating the path.

### The Strategy
We will implement a highly conservative, surgically targeted fix that leaves the core LLM orchestration intact while eliminating these three failure modes:
1. **Python Text Generation (`build_trip_plan`):** We replace `build_trip_plan` with a direct Python implementation (`method="direct"`). Given a route and constraints, it computes exact deterministic date math (eliminating off-by-one errors) and outputs the perfect `Arriving in {City}` string. This also reduces pipeline cost by roughly 40%.
2. **Python Regex Flight Fallback (`fix_flights_in_constraints`):** We intercept the constraints before routing and use a simple regex on the original problem text to perfectly overwrite the `flights` graph, completely curing the unidirectional flight hallucination.
3. **Tightened Prompt Rules (`find_valid_route`):** We add three specific rules to the routing prompt to strongly discourage dropping cities and to clarify that time windows explicitly lock in the arrival/departure dates.
