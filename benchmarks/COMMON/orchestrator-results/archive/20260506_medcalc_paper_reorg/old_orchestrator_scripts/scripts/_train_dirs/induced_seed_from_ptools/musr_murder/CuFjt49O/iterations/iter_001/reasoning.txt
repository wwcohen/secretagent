An analysis of the failure cases reveals a critical flaw in the generated orchestration logic: the `answer_question_workflow_orchestrated_seed` function evaluates suspects sequentially and returns the **first** suspect that gets a `'1'` from the LLM. 

Because the `_determine_if_killer_impl` tool evaluates each suspect in isolation, it frequently hallucinates a `'1'` for "ghost" suspects (distractors who are only mentioned once in the introductory sentence) simply because there is no evidence clearing them. If the ghost suspect happens to be `choices[0]`, the pipeline early-exits and returns them, completely ignoring the primary suspect who actually has the evidence.

To fix this conservatively without blowing up working cases:
1. **Evaluate All Suspects**: Instead of early-exiting, we will collect scores for all choices.
2. **Prompt Injection**: We append a strict instruction to the narrative to prevent the LLM from returning `'1'` for barely-mentioned suspects.
3. **Robust Tie-Breaker**: If the LLM returns `[0, 0]` or `[1, 1]`, we fall back to selecting the suspect with the highest mention count in the narrative, which acts as a very strong prior in this dataset.
4. **Bypass Unnecessary LLM Call**: We return the integer index directly instead of invoking `extract_index`, effectively removing an unnecessary LLM step and reducing cost/latency.
