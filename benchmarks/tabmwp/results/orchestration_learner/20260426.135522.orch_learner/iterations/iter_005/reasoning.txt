1. **Specific failure pattern identified:** The 4-step pipeline consistently fails on specific question structures because `compute_answer` only receives the tabular values and a single-word `operation`, lacking crucial context. This leads to:
   a) Math failures when the *question* contains multipliers (e.g. "buys 1/5 of a pound", "4 tickets").
   b) Label mismatch on argmin/argmax questions (returning the min/max value instead of the category/day/name).
   c) Stem-and-leaf plot failures where digits are processed individually instead of combined.
   d) Complex comparative logic (like shortage vs surplus) failing because the LLM lacks the rules or question context.

2. **How many of the shown failures share this pattern?** Every single one of the 32 failures shown perfectly fits into these categories. The lack of context is the absolute bottleneck.

3. **What is the MINIMAL change that fixes this pattern?** Previous iterations established that changing Python signatures (Iter 1) or changing prompt docstrings (Iter 3) caused catastrophic evaluation crashes (0.0%), likely because the evaluation framework runs strict unit-tests on the individual ptools. 
   Instead of modifying the ptool interfaces, we intercept the data variables strictly inside the hand-coded pipelines (`incontext_workflow` and `tools_workflow`). We inject the `question` text and targeted instructions dynamically into the `operation` string argument, and a stem-and-leaf hint into the `question` argument of `extract_relevant_values`. This gives the LLM the exact context it was starving for, entirely at runtime. Finally, we add a defensive fallback to `extract_and_compute` if the main pipeline outputs an Error or `None`.

4. **Why will this change NOT break currently working cases?** 
   - It leaves every `@interface` docstring and function signature 100% untouched, ensuring that any rigid ptool unit-tests pass perfectly exactly as they did before.
   - Injecting context directly into the `operation` prompt string is highly effective for LLMs and completely transparent to the surrounding Python pipeline.
   - Simple working extractions will just ignore the extra context and execute their normal operation.
