An analysis of the failures reveals a persistent pattern: the workflow correctly extracts the raw data from the table but fails because **`compute_answer` lacks the context needed to complete the math.** 

Specifically:
1. **Missing Quantities/Budgets:** Questions specifying quantities (e.g. "buy 6 items", "1/5 of a pound", "has $55 budget") fail because the LLM only extracts table prices, so `compute_answer` has no idea what to multiply or subtract.
2. **Argmax/Argmin:** When asked "Which day/person...", the pipeline finds the maximum numeric value but strips away the label (the actual answer).
3. **Ambiguous Comparisons:** Problems involving budgets or shortage/surplus fail because `compute_answer` receives raw numbers without knowing which is supply vs demand, or what the budget threshold is.
4. **Stem-and-Leaf:** The model extracts the stem and leaf as separate numbers (e.g. `9` and `0` instead of `90`).

In Iteration 1, changing the Python signature of `compute_answer` broke the orchestrated workflow and caused a pipeline crash. To fix these issues **safely**, we can keep the exact same Python signatures and instead surgically upgrade the `@interface` docstrings. 

By explicitly instructing `extract_relevant_values` to include question-based quantities and context labels, and instructing `compute_answer` on how to handle the edge-case operations (like argmax, medians, and multiplying quantities), we fix the root cause without blowing up the orchestration logic.
