1. **What SPECIFIC failure pattern did you identify?**
   In the failure cases, the LLM systematically misidentifies the killer due to three interacting issues:
   a) **Ghost Hallucinations**: It hallucinated evidence for "ghost" suspects who are barely mentioned (e.g., only in the first sentence).
   b) **Missed Exonerations**: It failed to recognize dataset-specific exonerating clues (Alibis, Disguises/Impersonations, or Fake Evidence like red paint).
   c) **Flawed Tie-Breaker**: When the LLM *did* correctly realize the Main suspect was innocent and returned `[0, 0]`, the original pipeline's tie-breaker fell back to selecting the suspect with the `max_count` (the highest number of mentions). This mathematically undid the LLM's correct deduction and forced it to predict the innocent Main suspect anyway.

2. **How many of the shown failures share this pattern?**
   All 17 failures. In cases where the expected killer is the ghost suspect (ex046, ex074, ex002, ex053), the LLM either missed the exonerating clue or the `[0, 0]` tie-breaker failed. In cases where the main suspect is guilty (ex039, ex045), the LLM hallucinated a '1' for the ghost suspect and the current prompt made it too hesitant to assign '1' to the main suspect.

3. **What is the MINIMAL change that fixes this pattern?**
   - **Ghost Suspect Short-Circuit**: Skip the LLM call entirely if a suspect is mentioned $\le 1$ time, assigning them a '0'. This eliminates hallucinations and halves API costs.
   - **Fix the `[0, 0]` Tie-Breaker**: If both scores are `0`, it means the LLM successfully identified that the Main suspect is innocent (due to an alibi or disguise). By boolean elimination, the true killer *must* be the Ghost suspect. We fix the tie-breaker to return `min_count` in this specific scenario.
   - **Refine the Prompt**: Replace the vague "strong evidence" prompt with explicit instructions tailored to the dataset's rules (Weapon Link = 1; Alibi/Disguise/Fake Evidence = 0).
   - **Robust Parsing**: Add string parsing that handles cases where the LLM might output text like "0. They have an alibi."

4. **Why will this change NOT break currently working cases?**
   Unlike Iteration 2 (which added the short-circuit but left the `max_count` tie-breaker broken, effectively failing all alibi cases), this solution mathematically aligns the Python fallback logic with the boolean structure of the dataset. The LLM is now highly directed to evaluate only the *substantive* suspect for specific clues, while the fallback logic perfectly captures the process of elimination.
