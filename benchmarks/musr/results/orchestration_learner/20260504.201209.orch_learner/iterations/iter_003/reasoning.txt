1. What SPECIFIC failure pattern did you identify?
The pipeline fails on complex MuSR logic puzzles where the true murderer is obscured by classic mystery tropes: ALIBIS (the suspect with the weapon is observed asleep elsewhere) or DISGUISES (the obvious suspect's hair/eye color changed, indicating an impostor framed them).

2. How many of the shown failures share this pattern?
All of them. For instance, in ex046, Lucy has an alibi (she went to bed and was observed by John), so Clara must be the killer. In ex002, Paige's hair and eye color changed, indicating Marianne was impersonating her. Because the current prompt evaluates suspects independently without acknowledging these tropes, the LLM takes the most literal path, incorrectly blaming the framed suspect and clearing the true killer.

3. What is the MINIMAL change that fixes this pattern?
Update the prompt enhancement injected into `_determine_if_killer_impl` to explicitly list the three dataset-specific deductive tropes (Means, Alibis, Disguises) and require the LLM to output a strict string like 'RESULT: 1' or 'RESULT: 0'. We also pass the names of the *other* suspects in the prompt so the LLM can reason comparatively, even though it evaluates one suspect at a time.

4. Why will this change NOT break currently working cases?
This replaces Iteration 2's failed attempt. Iteration 2 hardcoded Python rules (`count <= 1`) and forced the LLM to *only* look at who owned the weapon, completely destroying accuracy on Alibi/Disguise cases. This new approach does not skip any Python logic; instead, it empowers the LLM by defining the rules of the logic puzzle. The parsing is strictly backward-compatible (falling back to `'1' in is_killer` if the strict format isn't followed), ensuring safe handling of edge cases.
