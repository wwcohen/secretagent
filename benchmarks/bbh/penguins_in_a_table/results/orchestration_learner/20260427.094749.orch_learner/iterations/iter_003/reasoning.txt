An analysis of the failures reveals two primary issues causing both training errors and the large drop in evaluation set accuracy in the previous iteration:

1. **Over-filtering Multiple Tables (`valid.004`)**: In the previous iteration, `analyze_input` was instructed to always extract a *single* table. If the question was ambiguous, it defaulted to the immediately preceding table. While this fixed ambiguous single-table references, it completely broke cases where the question was explicitly about **all** tables (e.g., "How many animals are listed in the tables?"). The LLM would discard the first table, leading to an incorrect count.
2. **Lack of Strict CoT in Complex Comparisons (`valid.019`)**: When asked questions requiring relative comparisons (e.g., "older than Bernard and younger than Gwen"), `answer_question` attempted zero-shot evaluation without explicitly resolving the reference values first, leading to simple arithmetic/logic failures.

### The Fix
- Update `analyze_input` to explicitly handle plural table questions by instructing the LLM to combine their data rows into a single merged table.
- Enforce a strict Step-by-Step template in `answer_question` so it explicitly writes out reference values and evaluates rows line-by-line (e.g., True/False).
- Emphasize to `choose_response` that the answer will now contain verbosely structured reasoning, so it should only parse the final conclusion.
