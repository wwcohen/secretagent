An analysis of the execution traces and previous iterations reveals two primary issues causing failures:

1. **Python Evaluation Errors**: The LLM writes logically correct formulas, but embeds them in statements that cause `SyntaxError`s during Python's `eval()` step. Specifically, trailing results (e.g., `100 / 5 = 20`) or descriptive strings (e.g., `(in millions)`) crash the script, forcing the workflow to fall back to less reliable text extraction. Attempting to blanket-strip text units like "million" in prior iterations caused incorrect scale evaluations (e.g. `1.2 billion / 500 million` evaluated as `1.2 / 500`). 
2. **Percentage Format Mismatches**: When the system inevitably falls back to the text extractor, it frequently extracts answers with `%` signs (e.g. `55.07%`). However, the `FinQA` dataset strictly encodes percentages as decimal proportions in its golden labels (e.g. `-0.5507`, `0.00563`, `0.06507`).

### Targeted Changes:
- **Bulletproof Formula Parser (`_extract_formula`)**: Upgraded to safely find the mathematical core of the extracted formula string. Instead of failing on strings like `Percentage Change = (200 - 100) / 100 = 1.0`, the parser splits by `=` and selects the longest substring containing numbers and math operators. It also cleans trailing descriptive parentheses like `(in millions)`. 
- **Safe Percentage Evaluation (`answer_finqa_workflow`)**: Safely replaces `%` with `/100` *before* execution so natural mathematical evaluation correctly outputs the expected decimal proportion natively (e.g. `50%` executes as `0.5`).
- **Post-Processing Coercion (`_clean_percentage`)**: Intercepts the fallback LLM text. If the fallback correctly extracted `55.07%`, we mathematically scale it down to `0.5507`, perfectly satisfying FinQA's golden evaluation format. 
- **LLM Refinement (`extract_final_number`)**: Updated the docstring examples to encourage the LLM to output the decimal scale for percentages directly during the fallback phase.
