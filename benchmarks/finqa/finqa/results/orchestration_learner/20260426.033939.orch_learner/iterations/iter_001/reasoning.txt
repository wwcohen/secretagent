An analysis of the execution traces reveals a significant failure pattern: the pipeline often successfully generates a reasoning plan, but fails during the Python `compute` step because the `Formula` string contains artifacts like variable assignments (`revenue = ...`), inline evaluations (`... = 0.05`), or textual symbols (`%`, `million`).

When `compute` encounters these artifacts, it throws a syntax error and the workflow is forced to fall back to the purely LLM-based `extract_final_number(plan)`. By making our formula extraction and evaluation slightly more robust, we can guarantee exact mathematical evaluation in many cases that currently fail over to the unreliable fallback.

### Targeted Changes Made:
1. **Prompt Clarity (`extract_reasoning_plan`)**: Instructed the LLM explicitly to output only pure mathematical expressions without trailing equations or units.
2. **Robust Regex Extraction (`_extract_formula`)**: Upgraded the formula parser to strip common non-executable artifacts:
   - Removes leading variable assignments (e.g., `"target = 10 / 5"` → `"10 / 5"`).
   - Removes trailing evaluations (e.g., `"10 / 5 = 2"` → `"10 / 5"`).
   - Safely removes common written units (`million`, `billion`).
3. **Safe Evaluation (`compute`)**: 
   - Pre-processes `%` by replacing it with `/100` so that percentages don't cause modulo syntax errors in Python's `eval`.
   - Added a safe floating-point rounding fix (`1e-9`) before coercing floats into integers so `eval("3.3 * 100")` returns `"330"` rather than missing the int-check due to `330.00000000000006`.
