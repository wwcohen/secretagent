1. **Specific Failure Pattern:** The pipeline frequently fails during the Python `compute` step because the `Formula` string generated by the LLM contains text artifacts like trailing results (`= 0.05`), text units (`million`, `dollars`), or percent signs (`%` instead of `/100`). When `compute` throws a syntax error, the workflow falls back to an unstructured LLM guess (`extract_final_number`), which frequently hallucinates the final arithmetic or incorrectly outputs strings formatted with `%` when the dataset strictly expects a decimal (e.g. `0.06507`).
2. **Impact:** Over 15 of the visible failures (e.g. `valid.00255`, `valid.00460`, `valid.00183`, `valid.00533`) show the model failing with a percentage format or incorrect hallucinated math where a precise decimal float is expected.
3. **Minimal Fix:** 
   - Update `_extract_formula` to strictly drop everything after the first `=`. This perfectly handles the common `Formula: 100 / 5 = 20` pattern.
   - Update `compute` to replace `%` with `/100`. This allows natural percentage execution, natively outputting the dataset-expected decimals (e.g. `50%` evaluates to `0.5`). 
   - Update `compute` to use regex to silently strip all non-math alphabetic identifiers (e.g. `million`, `in`, `dollars`) from the formula string prior to Python `eval`. This safely handles expressions like `64 million / 549 million` by reducing it to `64 / 549`.
4. **Why it won't break working cases:** Unlike Iter 3, this doesn't replace "million" with `1e6` (which ruined questions asking for answers "in millions"). Unlike Iter 2, it does not alter the LLM prompt, avoiding prompt drift. Any formula that worked before will still work because math-safe functions (`abs`, `round`, `min`, `max`, `sum`) and standard Python syntax are deliberately retained.
