## Strategy Priorities (most effective to least)

NOTE: The model is FIXED. Do NOT propose upgrade or downgrade operators.
Focus only on code, prompt, and workflow improvements.

1. **ACCURACY FIRST**: If accuracy < target, focus on accuracy improvement:
   - evolve: rewrite ptool prompts/implementations using LLM evolution
   - swap_strategy: change method (e.g. simulate -> prompt_llm with custom template,
     or simulate -> direct with Python code + LLM fallback)
   - repair: fix error-prone ptools with better error handling
   - route: add conditional dispatch for different input types

2. **Python over LLM**: If a ptool's task can be solved reliably with Python code,
   swap_strategy to "direct" with LLM fallback is ALWAYS better than keeping an
   LLM call. Python code is free, fast, and deterministic. Keep the LLM as a
   fallback for edge cases.

3. **WORKFLOW IMPROVEMENTS**:
   - restructure: reorder calls (cheap first), add early exits, cache repeated calls
   - expand: break expensive multi-step ptools into focused sub-steps
   - induce: add lookup tables or shortcut logic for common patterns

## How to Read Profiling Data

IMPORTANT: "errors" in profiling data means EXCEPTIONS (crashes), not wrong answers.
A ptool can have 0 errors but still produce wrong answers (low accuracy).
Focus on accuracy improvement even when error counts are zero.

- **in_correct/in_wrong**: Shows what fraction of correct vs incorrect cases used
  this ptool. If in_wrong >> in_correct, this ptool may be CAUSING failures.
  If in_correct >> in_wrong, the ptool is doing well. If both are ~100%, the
  ptool is called universally and isn't a differentiator.
- **TOKEN_SATURATED**: If shown, output is hitting the token limit and may be
  truncated. This means the LLM is generating garbage after the limit — consider
  evolve to make the prompt request shorter output, or expand to split the task.
- **cost_frac**: What % of total pipeline cost this ptool uses. High cost_frac
  ptools are candidates for swap_strategy to direct (Python) with LLM fallback.
- **Pipeline code**: When shown, this is the actual Python code being executed.
  Use it to reason about what specific code changes would improve accuracy.

## Considerations

- Which ptools correlate with wrong answers? (in_wrong is high -> evolve, repair)
- Which ptools consume disproportionate cost? (-> swap_strategy to direct, restructure)
- Could a ptool be replaced by Python code + LLM fallback? (-> swap_strategy to direct)
- Could rewriting a ptool's prompt/implementation help? (-> evolve)
- Are there patterns in which cases succeed vs fail? (-> route, induce)
- Could an expensive ptool be split into cheaper focused steps? (-> expand)
- Could reordering calls or adding early exits help? (-> restructure)
- Is token saturation causing truncated/garbage output? (-> evolve, expand)
