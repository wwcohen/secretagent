Orchestration Learner Research Findings

Companion page for the Gemini-only sweep. See the self-contained curve dashboard and Markdown results table.

Research Questions

  1. Does iterative orchestration learning improve eval accuracy over the benchmark's configured workflow?
  2. Does seeding from generated ptools provide a better starting point than the existing workflow?
  3. When the learner fails, is the failure due to search quality, code edit application, or benchmark/tooling behavior?

Findings

  • Useful gains appear when the supervisor creates concrete, evaluated code changes; the strongest wins are not uniformly distributed across benchmarks.
  • Seeded workflows can improve compositional tasks, but they also create brittle failures where generated orchestration starts from a broken tool path.
  • RuleArena airline, NBA, and tax ran both classes and all five learner attempts, but every post-baseline attempt was a no-op code edit. The reports show eval_accuracy: null for those attempts, and ptools_before.py equals ptools_after.py.
  • For RuleArena, the plotted flat gray markers mean attempted but not re-evaluated, not missing experiment execution. Best T/E is selected from actually evaluated points, so these no-ops do not become artificial best iterations.

Conclusions

The orchestration learner is promising on tasks where its edits are applied and evaluated, but the current pipeline needs explicit accounting for no-op supervisor outputs, edit-parse failures, and benchmark-specific execution failures. Treating those separately from accuracy rollbacks will make the next sweep more diagnostic.
