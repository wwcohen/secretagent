# Planned Experiments

## Round 1: Assessing Fixed Engineered Workflows with a Fixed Set of Engineered Tools/PTools

### Main result

The main result should have `expt_name` of `workflow`x
.  This is using a
handcoded workflow and a handcoded set of ptools (plus tools, if your
task has "real" Python tool).  You should store it under
`benchmarks/COMMON/results/BENCHMARK/SUBTASK/`.

**Notes:** Use `cli/expt.py` to run the experiment if possible.  If
it's not possible let me know why and we will see how we need to
extend it.  *It's important to have results generated in a uniform
way.*

**Example:**  for `BENCHMARK/SUBTASK` of `bbh/sports_understanding` the results are
generated by 

```
	uv run python -m secretagent.cli.expt run --interface ptools.are_sports_in_sentence_consistent evaluate.expt_name=workflow \
	  ptools.are_sports_in_sentence_consistent.method=direct \
	  ptools.are_sports_in_sentence_consistent.fn=ptools.sports_understanding_workflow
```

which binds the top-level interface
`are_sports_in_sentence_consistent` in ptools.py to a
workflow-implementing function, specifically `sports_understanding_workflow`.
The code for implementing this in ptools.py is:

```
@interface
def consistent_sports(sport1: str, sport2: str) -> bool:
  """Compare two descriptions of sports, and determine if they are consistent.

  Descriptions are consistent if they are the same, or if one is more
  general than the other.
  """

def sports_understanding_workflow(sentence:str) -> bool:
  """Handcoded workflow for are_sports_in_sentence_consistent.
  """
  player, action, event = analyze_sentence(sentence)
  player_sport = sport_for(player)
  action_sport = sport_for(action)
  result = consistent_sports(player_sport, action_sport)
  if event:
    event_sport = sport_for(event)
    result = result and consistent_sports(player_sport, event_sport)
  return result
```

### First ablation of main result: Drop the workflow, retain the ptools.

Discarding the handcoded workflow means using some other strategy to
plan a sequence of ptool/tool calls.  There are two you should try:
 * Program of thought
 * ReAct (via Pydantic AI's implementation)

These ablations should have `expt_name` of `pot` and `react`.

**Example:** for `BENCHMARK/SUBTASK` of `bbh/sports_understanding`
those results are generated by

```
pot:
	uv run python -m secretagent.cli.expt run --interface ptools.are_sports_in_sentence_consistent evaluate.expt_name=pot \
	  ptools.are_sports_in_sentence_consistent.method=program_of_thought \
	  ptools.are_sports_in_sentence_consistent.tools=[ptools.analyze_sentence,ptools.sport_for,ptools.consistent_sports]

react:
	uv run python -m secretagent.cli.expt run --interface ptools.are_sports_in_sentence_consistent evaluate.expt_name=react \
	  ptools.are_sports_in_sentence_consistent.method=simulate_pydantic \
	  ptools.are_sports_in_sentence_consistent.tools=[ptools.analyze_sentence,ptools.sport_for,ptools.consistent_sports]
```

### Second ablation of main result: Drop the workflow and the ptools

This ablation should have `expt_name` of `structured_baselines`.  For
`BENCHMARK/SUBTASK` of `bbh/sports_understanding` the results are
generated by

```
structured_baseline: 
	uv run python -m secretagent.cli.expt run --interface ptools.are_sports_in_sentence_consistent evaluate.expt_name=structured_baseline \
	  ptools.are_sports_in_sentence_consistent.method=simulate
```

Note the LLM is getting the same information as in `pot` and `react`

### Third ablation of main result: Drop the workflow, ptools, and structured top-level prompt

The structured baseline is a very weird prompting method - the prompt
the LLM gets asks it to simulate this Python stub.  

```
def consistent_sports(sport1: str, sport2: str) -> bool:
  """Compare two descriptions of sports, and determine if they are consistent.

  Descriptions are consistent if they are the same, or if one is more
  general than the other.
  """
```

The last baseline is to get rid of the weirdness and use an
*equivalent but "normal" prompt.*  It's not always obvious what this
means - for one thing, "normal" prompts produce strings, which might
be harder to evaluate than the structured outputs produced by all the
other baselines.

**Example:** For `sports_understanding` I used a zero-shot prompt that
corresponded closely to the top-level interface instructions, followed
by a one-step LLM-based cleanup to standardize the output.  (I tried
to do it without the cleanup step but that turned out to be a pretty
weak baseline).

```
@implement_via('prompt_llm', prompt_template_file='prompt_templates/zeroshot.txt')
def zeroshot_are_sports_in_sentence_consistent(sentence: str) -> str:
  ...

@implement_via('simulate')
def convert_llm_output_to_true_or_false(llm_output: str) -> bool:
  """Given an llm's output, approximate the intended answer as bool.
  """
  ...

def zeroshot_unstructured_workflow(sentence:str) -> bool:
  """Workflow for using a zero-shot prompt and coercing the type to bool.

  To run the zeroshot unstructured model, bind this to the
  implementation of 'are_sports_in_sentence_consistent'.
  """
  llm_output = zeroshot_are_sports_in_sentence_consistent(sentence)
  return convert_llm_output_to_true_or_false(llm_output)
```
where the zeroshot.txt file is
```
Determine plausibility of a sports-related sentence.

Specifically, determine if the sports associated with the player,
action and event in a sentence are all consistent with each other.

Your answer should be either 'True' or 'False'.  Use the following format:

<answer>
YOUR ANSWER
</answer>

The sentence is: $sentence
```
