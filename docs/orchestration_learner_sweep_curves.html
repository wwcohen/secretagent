Research Questions & Findings

Why some RuleArena charts look flat: RuleArena airline, NBA, and tax all ran both experiment classes with a baseline plus five learner attempts. The attempts were no-ops: the saved ptools_before.py and ptools_after.py files are identical, so no train/eval result directories were produced after baseline. The dashboard shows those attempts as gray carried-forward markers rather than hiding them.

Question: Does iterative orchestration learning improve held-out eval accuracy over the configured workflow, and does seeding from generated ptools help?

Finding: Gains are concentrated in tasks where the supervisor produced concrete, evaluated code changes. Seeded runs helped several compositional tasks, but also produced brittle zero-accuracy failures on some domains. RuleArena is currently mostly a failure-to-edit signal rather than a measured search curve.

Conclusion: The learner can discover useful orchestration edits, but benchmark-specific adapter/tooling failures and no-op supervisor responses are first-order blockers. Future sweeps should track no-op/edit-parse failures separately from evaluated rollbacks.

Analysis

Key insight: The learner is not just "improving prompts" — it is acting as a structural repair loop. When it can produce concrete code edits and those edits execute, train and eval often move together, suggesting it is sometimes discovering real workflow structure rather than simply overfitting train examples.

Hypothesis

Seeding from ptools helps because it gives the learner a manipulable decomposition earlier. The learner performs best when the task has separable latent steps: parse constraints, compute deterministically, verify a candidate, normalize an answer, etc. This explains why NaturalPlan, geometric shapes, MUSR (object/team/murder), MedCalc, and penguins show meaningful gains.

Cautions: systems failures, not scientific failures

Many "failures" are not scientific failures of the idea — they are orchestration-system failures:

  • RuleArena mostly produced no-op edits: baseline plus five attempts, but no real code delta.
  • TabMWP has a suspicious final-eval collapse to 0.0% despite 50.0% per-iteration eval; final eval reloads the evolved ptools after the one-time setup hook has already populated the table store, so the reload clears the in-memory tables.
  • Some eval-best edits were rolled back because train was the selection objective, so the learner can miss eval-favorable variants.

Conclusion

Automated orchestration learning works when the edit loop is healthy and the task admits reusable structure. The next bottleneck is not model capability alone; it is edit validity, no-op detection, final-eval reliability, and selection criteria.

Honest assessment: This is promising enough to justify a second sweep, but the paper/result should emphasize "conditional capability with systems bottlenecks," not "fully reliable automatic workflow induction" yet.
